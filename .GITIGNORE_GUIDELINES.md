# TGSDK Git Ignore Guidelines

## Overview

This document explains the `.gitignore` strategy used in the TGSDK Laravel package project.

## Philosophy

Our `.gitignore` follows these principles:

1. **Track source code, not generated files** - Only commit files that developers create manually
2. **Environment isolation** - Keep local configurations separate from shared code
3. **Security first** - Never commit secrets, keys, or credentials
4. **Cross-platform friendly** - Ignore OS-specific files
5. **Package-aware** - Exclude Laravel package-specific runtime files

## What's Ignored

### Critical Exclusions (Never Commit)

- `.env` - Environment variables and secrets
- `vendor/` - Composer dependencies (use `composer install`)
- `node_modules/` - NPM dependencies
- `storage/` - Runtime files (logs, cache, sessions)
- `*.log` - All log files
- `*.key` - Encryption keys
- `*.session` - Telegram session files

### Development Files

- IDE configurations (`.idea/`, `.vscode/`)
- Editor swap files (`*.swp`, `*.swo`)
- OS metadata (`.DS_Store`, `Thumbs.db`)
- Test cache files (`.phpunit.result.cache`)

### Python Worker

- Virtual environments (`venv/`, `.venv/`)
- Compiled Python files (`*.pyc`, `__pycache__/`)
- Session files (`sessions/`)

## What's Tracked

### Source Code

- ✅ All PHP files in `src/`, `app/`, `config/`, `routes/`
- ✅ Migration files in `database/migrations/`
- ✅ Test files in `tests/`
- ✅ Documentation in `docs/`
- ✅ Configuration templates

## Setup Instructions

### For New Developers

1. Clone the repository
2. Copy `.env.example` to `.env`
3. Run `composer install`
4. Run `npm install` (if applicable)
5. Generate app key: `php artisan key:generate`
6. Set up database and run migrations

### For CI/CD

Ensure your build process:
1. Creates necessary directories
2. Sets environment variables
3. Installs dependencies
4. Generates application key
5. Does NOT rely on committed secrets

## Common Questions

### Q: Should I commit `composer.lock`?

**A:** For packages, it's optional. We allow flexibility by not committing it, but you can if you want strict version control.

### Q: What about configuration files?

**A:** Track base configs, ignore environment-specific overrides. Use `config/local/` for local testing.

### Q: Can I track some logs for debugging?

**A:** No. Logs should never be committed. Use logging services or share logs privately when debugging.

### Q: How do I handle database credentials?

**A:** Use `.env` file with `DB_*` variables. Each developer has their own local database.

### Q: What about test fixtures?

**A:** Small fixtures can be tracked. Large files (>1MB) should be excluded or stored externally.

## Best Practices

1. **Review before committing** - Always check `git status` before adding files
2. **Use `.env.example`** - Provide a template with placeholder values
3. **Document exceptions** - If you need to commit something unusual, document why
4. **Keep `.gitignore` updated** - Add new patterns as the project evolves
5. **Test fresh clone** - Periodically verify new developers can setup easily

## Security Notes

Never commit:
- API keys (Telegram Bot Token, etc.)
- Database passwords
- Encryption keys
- OAuth credentials
- AWS/cloud credentials
- SSH private keys
- SSL certificates with private keys

If accidentally committed:
1. Remove the file immediately
2. Rotate the compromised credentials
3. Clean git history if necessary
4. Document the incident
