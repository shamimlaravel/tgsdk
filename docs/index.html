Overview

Laravel Telegram Hybrid Storage

A Laravel 12 package that implements a custom filesystem driver backed by Telegram channels. Files are uploaded asynchronously through a Python worker (Pyrogram) via a Redis queue, and served back through a streaming proxy with optional CDN support.

Repository: https://github.com/shamimlaravel/tgsdk.git

Laravel Storage API

Use Storage::disk('telegram')->put(), get(), url(), exists() seamlessly

Telegram-backed Storage

Files stored as messages in Telegram channels via MTProto (Pyrogram)

Async Upload Pipeline

Laravel enqueues to Redis; Python worker handles Telegram upload

Unlimited File Size

Intelligent chunking splits files beyond Telegram's 2GB/4GB limits

Channel Rotation

Distributes uploads across multiple channels (round-robin, least-used)

Security

Chunk encryption (AES-256-GCM) and integrity verification (SHA-256)

Component Responsibilities

TelegramStorageAdapter

Implements Flysystem's FilesystemAdapter interface. Handles file operations (put, get, delete, exists) and coordinates with other components.

  • Interfaces with Laravel Storage facade
  • Manages file metadata in database
  • Triggers upload jobs to Redis queue
  • Handles streaming downloads

ChannelRotator

Selects the optimal Telegram channel for each upload based on configured strategy.

  • Round-robin distribution
  • Least-used selection
  • Capacity-aware routing
  • Channel health monitoring

ChunkManager

Splits large files into manageable chunks and reassembles them on download.

  • File chunking algorithm
  • Chunk ordering and tracking
  • Reassembly logic
  • Compression/encryption support

IntegrityVerifier

Ensures data integrity through checksums and hashes.

  • SHA-256 checksum generation
  • Chunk-level verification
  • File-level verification
  • Tamper detection

Python Worker (SessionPool)

Manages multiple Pyrogram sessions for parallel uploads.

  • Session lifecycle management
  • Connection pooling
  • Rate limiting
  • Error recovery

Python Worker (Uploader)

Processes upload queue and sends files to Telegram.

  • Queue consumption
  • File upload to Telegram
  • Callback notifications
  • Retry logic

Detailed Workflow Steps

1

Application Uploads File

Storage::disk('telegram')->put('path/file.zip', $contents);

The application calls the storage facade to upload a file. The adapter creates a database record and determines if chunking is needed.

2

Job Enqueued to Redis

An upload job is created with file metadata and pushed to the Redis queue. The application returns immediately without waiting for the actual upload.

3

Python Worker Processes Queue

The worker polls the Redis queue, retrieves upload jobs, and selects an available session from the pool.

4

File Uploaded to Telegram

Using Pyrogram, the worker uploads the file (or chunks) to the selected Telegram channel(s) via MTProto API.

5

Callback to Laravel

After successful upload, the worker sends a callback to Laravel with message IDs and file information.

6

Database Updated

Laravel verifies the callback signature, updates the file record with Telegram message IDs, and fires completion events.

Installation

Step 1: Install Package

composer require shamimstack/tgsdk-laravel-telegram-hybrid-storage

Step 2: Publish Configuration

php artisan vendor:publish --tag=telegram-storage-config
php artisan vendor:publish --tag=telegram-storage-migrations
php artisan migrate

Step 3: Configure Filesystem

Add to config/filesystems.php:

'disks' => [
    // ...
    'telegram' => [
        'driver' => 'telegram',
    ],
],

Step 4: Set Environment Variables

# Telegram API credentials
TELEGRAM_API_ID=your_api_id
TELEGRAM_API_HASH=your_api_hash
TELEGRAM_BOT_TOKEN=your_bot_token
TELEGRAM_SESSION_NAME=telegram_storage

# Redis
TELEGRAM_STORAGE_REDIS_CONNECTION=default
TELEGRAM_STORAGE_REDIS_QUEUE=telegram_upload_queue

# Worker callback
TELEGRAM_STORAGE_CALLBACK_URL=https://your-app.com/telegram-storage/callback
TELEGRAM_STORAGE_CALLBACK_SECRET=your_hmac_secret

Step 5: Setup Python Worker

cd python-worker
cp .env.example .env
# Edit .env with your credentials
pip install -r requirements.txt
python worker.py

Or use Docker:

docker build -t telegram-worker .
docker run --env-file .env telegram-worker

Usage Examples

Basic File Operations

use Illuminate\Support\Facades\Storage;

// Upload a file
Storage::disk('telegram')->put('documents/report.pdf', $fileContents);

// Upload from stream
Storage::disk('telegram')->putStream('videos/clip.mp4', fopen('/path/to/file', 'r'));

// Check existence
$exists = Storage::disk('telegram')->exists('documents/report.pdf');

// Get contents
$contents = Storage::disk('telegram')->get('documents/report.pdf');

// Get URL
$url = Storage::disk('telegram')->url('documents/report.pdf');

// Get metadata
$size = Storage::disk('telegram')->size('documents/report.pdf');
$mime = Storage::disk('telegram')->mimeType('documents/report.pdf');

// Delete
Storage::disk('telegram')->delete('documents/report.pdf');

Event Handling

use Shamimstack\Tgsdk\Events\TelegramUploadCompleted;
use Shamimstack\Tgsdk\Events\TelegramUploadFailed;

// In EventServiceProvider
Event::listen(TelegramUploadCompleted::class, function ($event) {
    Log::info("File uploaded: {$event->path}", ['file_id' => $event->fileId]);
});

Event::listen(TelegramUploadFailed::class, function ($event) {
    Log::error("Upload failed: {$event->path}", ['error' => $event->error]);
});

Available Events

Event Triggered When
TelegramUploadQueued File upload job dispatched to Redis
TelegramUploadCompleted Upload confirmed successful
TelegramUploadFailed Upload failed after all retries
TelegramChunkCompleted Individual chunk upload confirmed
TelegramChunkFailed Individual chunk upload failed
TelegramUploadStalled Upload detected as stalled
TelegramFileDeleted File record deleted

Configuration

Configuration Options

Key configuration options available in config/telegram-storage.php:

Option Default Description
channels [] List of Telegram channel IDs for storage
rotation_strategy round-robin Channel selection: round-robin, least-used, capacity-aware
chunk_threshold 1950000000 File size (bytes) above which chunking activates
chunk_size 1950000000 Size of each chunk
chunk_compression false Enable gzip compression per chunk
chunk_encryption false Enable AES-256-GCM encryption per chunk
download.signed_urls false Enable signed download URLs
download.url_ttl 3600 Signed URL TTL in seconds
cdn.enabled false Enable CDN URL prefix

API Reference

Core Classes

TelegramStorageAdapter

Namespace: Shamimstack\Tgsdk\TelegramStorage

Description: Main adapter implementing Flysystem's FilesystemAdapter interface

Methods:

write($path, $contents, Config $config) Write file contents to Telegram
writeStream($path, $resource, Config $config) Write file from stream to Telegram
read($path) Read file contents from Telegram
readStream($path) Read file as stream from Telegram
delete($path) Delete file from Telegram and database
fileExists($path) Check if file exists
directoryExists($path) Check if directory exists
getSize($path) Get file size in bytes
getMimeType($path) Get file MIME type
getUrl($path) Get download URL (optionally signed)

ChannelRotator

Namespace: Shamimstack\Tgsdk\TelegramStorage

Description: Selects optimal Telegram channel based on rotation strategy

Methods:

selectChannel() Select next available channel
markUsed($channelId) Mark channel as used
getStrategy() Get current rotation strategy

ChunkManager

Namespace: Shamimstack\Tgsdk\TelegramStorage

Description: Manages file chunking and reassembly

Methods:

shouldChunk($size) Determine if file should be chunked
createChunks($fileId, $size) Create chunk records for file
reassemble($fileId) Reassemble file from chunks
compressChunk($data) Compress chunk data
encryptChunk($data) Encrypt chunk data

IntegrityVerifier

Namespace: Shamimstack\Tgsdk\TelegramStorage

Description: Verifies data integrity using checksums

Methods:

generateChecksum($data) Generate SHA-256 checksum
verifyChecksum($data, $checksum) Verify data against checksum
generateFileHash($path) Generate file hash

CSS Architecture Principles

BEM Methodology

We follow the BEM (Block Element Modifier) methodology for naming CSS classes:

.block {}
.block__element {}
.block--modifier {}

// Example:
.sidebar {}                    // Block
.sidebar__nav-menu {}          // Element
.sidebar__nav-item--active {}  // Modifier

Mobile-First Approach

All styles are written mobile-first, then enhanced for larger screens:

// Base styles (mobile)
.component {
    padding: $spacing-4;
    font-size: $font-size-sm;
}

// Tablet and up
@include respond-to(md) {
    .component {
        padding: $spacing-6;
        font-size: $font-size-base;
    }
}

// Desktop and up
@include respond-to(lg) {
    .component {
        padding: $spacing-8;
    }
}

CSS Custom Properties

CSS custom properties (variables) are used for dynamic theming:

:root {
    --primary-color: #{$primary-color};
    --secondary-color: #{$secondary-color};
    --spacing-unit: #{$spacing-4};
    --border-radius: 8px;
}

.component {
    background: var(--primary-color);
    padding: var(--spacing-unit);
    border-radius: var(--border-radius);
}
